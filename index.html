Базовые понятия

Uniform Resource Locators (URL)

https://developer.mozilla.org/en-US/docs/Learn/Common_questions/What_is_a_URL

URL является одной из ключевых концепций в вебе. URL – это просто адрес какого-то уникального ресурса в интернете. Это может быть HTML файл, видео, CSS файл. Из чего состоит URL?

Структура

  • Первая часть – протокол. Может быть как http, так https. Браузер также может обрабатывать mailto и ftp.
  • Доменное имя, которое может содержать саб-домены и обязательный домен верхнего уровня: .ru, .com, .net, .org, etc. Доменное имя указывает на сервер, куда в конечном итоге будет обращаться браузер. Вместо доменного имени может быть IP адрес сервера.
  • Порт – если порт не указан, то используется 80 для HTTP протокола и 443 для HTTPS.
  • Путь к файлу – изначально это был путь к физическому файлу на сервере, а сейчас абстракция, которая обрабатывается сервером, после чего отдаются какие-либо данные.
  • Параметры – дополнительные параметры, которые записываются после вопроса и разделяются амперсандом. Имеют вид ключ=значение.
  • Якорь дает браузеру информацию как нужно отобразить содержимое. Например для страницы он указывает в какую часть страницы нужно проскролить окно. Нужно отметить, что якорь не шлется с запросом на сервер.

Виды URL

1. Абсолютный: https://www.google.com/search?q=test;

3. Релативный: ../search?q=test

2. Неявный протокол: //www.google.com/search?q=test

4. Неявный домен: /search?q=test

Domain Name System. Система доменных имён

DNS (Domain Name System «система доменных имён») — компьютерная распределённая система для получения информации о доменах. Чаще всего используется для получения IP-адреса по имени хоста (компьютера или устройства). Грубым языком – сервер, который имеет соответствие доменного имени и IP адреса.

TCP

Transmission Control Protocol (TCP, протокол управления передачей) — один из основных протоколов передачи данных интернета. Механизм TCP предоставляет поток данных с предварительной установкой соединения, осуществляет повторный запрос данных в случае потери данных и устраняет дублирование при получении двух копий одного пакета, гарантируя тем самым целостность передаваемых данных и уведомление отправителя о результатах передачи. Все ваши данные, которые вы хотели/хотите послать на сервер оборачиваются изначально в HTTP протокол, а после этого в TCP. Это работает для HTTP/1.1 и HTTP/2.0, но в HTTP/3.0 данный протокол планируют заменить на UDP. Иногда это спрашивают на собесах.

HTTP

Протокол прикладного уровня передачи произвольных данных. Это обычно преподается в универах, так что давайте и мы быстренько посмотрим что это такое в упрощенном виде. Это кстати иногда спрашивают на собесах. Что такое протокол прикладного уровня? Есть такое понятие как 7-уровневая модель ОСИ. Посредством данной модели различные устройства могут взаимодействовать в сети. Тут каждый уровень выполняет определенные действия для преобразования данных и манипуляции с адресами отправителя и получателя. Например прикладной уровень позволяет приложению обернуть ваши данные, которые вы хотите послать через интернет в HTTP протокол и отправить их дальше по модели ОСИ. Еще через несколько степов то, что вы хотите отправить оборачивается уже в TCP протокол, который уже будет дублировать ваши запросы или отклонять при таймаутах. Потом добавляется IP адрес источника и получателя, преобразование в электрические сигналы и отправка по проводам и как только вторая сторона получает сигнал все это же происходит заново. Давайте вернемся к HTTP. HTTP предполагает под собой, что есть потребитель данных и их поставщик. Другими словами клиент и сервер. Клиент инициирует запрос, а сервер производит необходимые действия для генерации ответа.

Структура

Каждое HTTP-сообщение состоит из трёх частей, которые передаются в указанном порядке:

  1. Стартовая строка.
  2. Заголовки.
  3. Тело сообщения.

Стартовая строка (Starting line)

Стартовая строка определяет тип сообщения.Разделяют на запрос/ответ:

Запрос содержит:

  1. Метод – тип запроса, одно слово заглавными буквами;
  2. URI – определяет путь к запрашиваемому документу;
  3. Версию.

Ответ содержит:

  1. Версию;
  2. Код состояния – код ответа, разберем какие могут быть чуть позже;
  3. Пояснение – текстовое короткое пояснение к ответу.

Заголовки (headers)

Это строки в HTTP-сообщении, содержащие пару параметр-значение. Позволяют прокидывать дополнительную информацию Все заголовки разделяются на четыре основных группы:

  • General Headers («Основные заголовки») — могут включаться в любое сообщение клиента и сервера;
  • Request Headers («Заголовки запроса») — используются только в запросах клиента;
  • Response Headers («Заголовки ответа») — только для ответов от сервера;
  • Entity Headers («Заголовки сущности») — сопровождают каждую сущность сообщения.

Тело сообщения (Message Body)

Тело HTTP-сообщения, если оно присутствует, используется для передачи тела объекта, связанного с запросом или ответом.

Методы

Последовательность из символов, указывающая на основную операцию над ресурсом.

1

GET

Используется для запроса содержимого указанного ресурса. С помощью метода GET можно также начать какой-либо процесс. Является идемпотентным. Идемпотентность – свойство объекта или операции при повторном применении операции к объекту давать тот же результат, что и при первом.
Примеры использования: получить список юзеров для сайта, получить какой-либо файл со скриптам, получить информацию про товар в каталоге. Тут нужно понимать, что GET только получает данные.

2

PUT

Применяется для загрузки содержимого запроса на указанный в запросе URI. Если по заданному URI не существует ресурс, то сервер создаёт его и возвращает статус 201 (Created). Если же был изменён ресурс, то сервер возвращает 200 (Ok) или 204 (No Content). Фундаментальное различие методов POST и PUT заключается в понимании предназначений URI ресурсов. Метод POST предполагает, что по указанному URI будет производиться обработка передаваемого клиентом содержимого. Используя PUT, клиент предполагает, что загружаемое содержимое соответствует находящемуся по данному URI ресурсу.
Примеры использования: обновить все данные о юзере.

3

POST

Применяется для передачи пользовательских данных заданному ресурсу. Не считается идемпотентным, то есть многократное повторение одних и тех же запросов POST может возвращать разные результаты.
Примеры использования: создать юзера, сохранить данные оплаты, добавить комментарий.

4

OPTIONS

Используется для определения возможностей веб-сервера или параметров соединения для конкретного ресурса. В ответ серверу следует включить заголовок Allow со списком поддерживаемых методов. Также в заголовке ответа может включаться информация о поддерживаемых расширениях. Обычно разработчик не отправляет эти запросы, а браузер это делает за него. Зачем и когда мы рассмотрим чуть позже. Есть и еще несколько методов, но мы рассмотрели основные, если хотите посмотреть что есть, можете загуглить ;)

5

PATCH

Аналогично PUT, но применяется только к фрагменту ресурса. Грубо говоря, когда пытаетесь изменить часть ресурса – юзайте PATCH, когда весь ресур – юзайте PUT. При разработке проекта программисты обычно договариваются использовать что-то одно: или PATCH, или PUT.

6

DELETE

Удаляет указанный ресурс.
Примеры использования: удалить юзера, удалить товар в каталоге.

Коды состояний

Код состояния отправляет сервер в ответ на запрос. Клиент узнаёт по коду ответа о результатах его запроса и определяет, какие действия ему предпринимать дальше. Код состояний представляет собой целое число из трёх цифр. Первая цифра указывает на класс состояния.

https://en.wikipedia.org/wiki/List_of_HTTP_status_codes

В HTTP обычно используются 5 классов:

  1. 1xx - Информирование о процессе передачи. Например 102 код - запрос принят, но на его обработку понадобится длительное время;
  2. 2xx - Информирование о случаях успешного принятия и обработки запроса клиента;
    1. 200 - OK - все хорошо
    2. 201 - Created - что-то создано
    3. 202 - Accepted - запрос был принят на обработку, но она не завершена
    4. 204 - No content - все ОК, но ответ не содержит тела
  3. 3xx - Сообщает клиенту, что для успешного выполнения операции необходимо сделать другой запрос;
  4. 4xx - Указание ошибок со стороны клиента;
    1. 400 - Bad request - плохие данные. Сервер обнаружил в запросе клиента синтаксическую ошибку
    2. 401 - Unauthorized - не авторизован. Для доступа к запрашиваемому ресурсу требуется аутентификация
    3. 403 - Forbidden - запрещено. Сервер понял запрос, но он отказывается его выполнять из-за ограничений в доступе для клиента к указанному ресурсу
    4. 404 - Not found - не найдено
    5. 405 - Method not allowed. Указанный клиентом метод нельзя применить к текущему ресурсу
    6. 409 - Conflict. Запрос не может быть выполнен из-за конфликтного обращения к ресурсу
    7. 429 - To many requests. Клиент попытался отправить слишком много запросов за короткое время, что может указывать, например, на попытку DDoS-атаки
  5. 5xx - Информирование о случаях неудачного выполнения операции по вине сервера.
    1. 500 - Internal server error
    2. 501 - Not implemented

CORS

https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS

Cross-origin resource sharing (совместное использование ресурсов между разными источниками).

Технология современных браузеров, которая позволяет предоставить веб-страницам доступ к ресурсам другого домена.

Для этого браузер лично шлет OPTIONS запрос, чтобы убедиться, может ли наш сайт обратиться к ресурсу другого домена или нет. Выделяются запросы, которые называют простыми. Если разработчик пытается послать простой запрос на другой домен, то браузер не будет посылать дополнительный OPTIONS запрос.

Браузер посылает дополнительный заголовок если запрос не удовлетворяет следующим требованиям:

  1. Метод: GET, HEAD, POST
  2. Кроме дефолтных заголовков браузера (список ниже) можно установить только следующие заголовки: Accept, Accept-Language, Content-Language, Content-Type
    1. Accept-Charset
    2. Accept-Encoding
    3. Access-Control-Request-Headers
    4. Access-Control-Request-Method
    5. Connection
    6. Content-Length
    7. Cookie
    8. Cookie2
    9. Date
    10. DNT
    11. Expect
    12. Host
    13. Keep-Alive
    14. Origin
    15. Referer
    16. TE
    17. Trailer
    18. Transfer-Encoding
    19. Upgrade
    20. Via
  3. Content-Type может содержать только следующие значения: application/x-www-form-urlencoded, multipart/form-data, text-plain
  4. Без ридблСтримов
  5. Нельзя устанавливать эвентов на XMLHttpRequestUpload

Давайте посмотрим как это работает:

В первом запросе мы просто сделали GET на другой ресурс без каких-то заголовков. Тут все хорошо и браузер не будет посылать дополнительного запроса.

Firefox/71.0

Accept-Language: en-us,en;q=0.5

Accept-Encoding: gzip,deflate

Server: Apache/2

Access-Control-Allow-Origin: *

Keep-Alive: timeout=2, max=100

Connection: Keep-Alive

Transfer-Encoding: chunked

Content-Type: application/xml

[…XML Data…]

Если это не простой запрос, то будет отправлен дополнительный (preflighted) запрос. Давайте рассмотрим следующий код:

const xhr = new XMLHttpRequest();
xhr.open('POST', 'https://bar.other/resources/post-here/');
xhr.setRequestHeader('X-PINGOTHER', 'pingpong');
xhr.setRequestHeader('Content-Type', 'application/xml');
xhr.onreadystatechange = handler;
xhr.send('<person><name>Arun</name></person>');

Здесь браузер пошлет дополнительный запрос, т.к. Используется нестандартный content-type и есть дополнительный небезопасный header

Лучше не юзать редиректы на preflighted request. Многие браузеры могут кинуть ошибку.Запрос с credentials.

По-дефолту запросы которые инициируются с javascript’а не добавляют данных с информацией аутентификации. Для этого обычно используются куки. Для этого нужно добавить флаг withCredentials в запрос, что даст доступ к кукам для запроса/ответа.

Для таких запросов, если он удовлетворяет требованию “простой запрос”, не пошлется OPTIONS запрос. Но если в ответе не будет заголовка Access-Control-Allow-Credentials, то браузер выкинет ошибку.

Что происходит, когда набираешь URL в строке ввода

Links

  • https://medium.com/@julianfrancor9/what-happens-when-you-type-a-url-in-your-browser-and-press-enter-50312e4f346c
  • https://developer.mozilla.org/en-US/docs/Learn/Common_questions/How_does_the_Internet_work
  • https://developer.mozilla.org/en-US/docs/Learn/Getting_started_with_the_web/How_the_Web_works
  1. Браузер проверяет свой кэш на наличия записи доменного имени и соответствующего IP адреса.
  2. раузер спрашивает операционную систему на наличие данного соответствия в файле hosts.
    1. C:\windows\system32\drivers\etc\hosts
    2. /etc/hosts
  3. сли домен нигде не закэширован и отсутствует в файле hosts, отправляет запрос к сетевому DNS-серверу. Как правило, это локальный роутер или DNS-сервер интернет-провайдера.
    1. Если интернет провайдер не знает доменного имени, то запрос отправляется к Root Server, который знает где располагается DNS сервер для домена верхнего уровня (e.g. .com).
    2. Браузер обращается напрямую к ДНС серверу домена верхнего уровня
    3. ДНС сервер отвечает, знает ли он IP адрес для домена, если там используется сабдомен, то мы переходим к пункту 5 начинаем заново.
    4. После всех итераций у браузера есть IP адрес нужного сервера и протокол соединения (HTTP, HTTPS), после чего браузер устанавливает TCP соединение с этим сервером. В зависимости от протокола, браузер обращается к 80 или к 443 порту, что соответствует HTTP и HTTPS протоколам соответственно. Поговорить про трехэтапное рукопожатие TCP/UDP и HTTP/HTTPs Клиентский компьютер отправляет сообщение ClientHello серверу со своей версией протокола TLS
    1. Сервер отвечает клиенту сообщением ServerHello, содержащим версию TLS и публичный сертификат сервиса, подписанный центром сертификации. Сертификат содержит публичный ключ, который будет использоваться клиентом для шифрования оставшейся части процедуры «рукопожатия» (handshake), пока не будет согласован симметричный ключ.
    2. Клиент подтверждает сертификат сервера с помощью своего списка центров сертификации. Если сертификат подписан центром из списка, то серверу можно доверять, и клиент генерирует строку псевдослучайных байтов и шифрует её с помощью публичного ключа сервера. Эти случайные байты могут быть использованы для определения симметричного ключа.
    3. Сервер расшифровывает случайные байты с помощью своего секретного ключа и использует эти байты для генерации своей копии симметричного мастер-ключа.
    4. Клиент отправляет серверу сообщение Finished, шифруя хеш передачи с помощью симметричного ключа.
    5. Сервер генерирует собственный хеш, а затем расшифровывает полученный от клиента хеш, чтобы проверить, совпадёт ли он с собственным. Если совпадение обнаружено, сервер отправляет клиенту собственный ответ Finished, также зашифрованный симметричным ключом.
    6. После этого TLS-сессия передаёт данные приложения (HTTP), зашифрованные с помощью подтверждённого симметричного ключа.
  4. Когда TCP соединение установлено, браузер шлет HTTP запрос к серверу, сервер отвечает каким-либо ответом. Схема, которую сделали на халбертонСкул. На ней выделили дополнительно с чего состоит бэкенд и как это может работать.
